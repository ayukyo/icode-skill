# 对抗分析模式（共享规则）

> 本文件是 icode 对抗分析的共享规则（分析师+3质疑者+裁决优先级+诚实降级）。步骤2 审查 issue、入口 log 根因分析，均复用本模式。各步骤文件只声明"分析对象"（issue/根因假设），其余对抗流程引用本文件。

## 为什么必须对抗

单视角结论存在确认偏误风险（自己提的假设，容易"看着像对就收"）。只有当独立一方尝试反驳却反驳不倒，结论才可信。未经对抗的结论视为"待验证"，不计入收敛判据。

## 角色分工

用 `Agent` 工具 spawn 3 个独立质疑者（**必须独立 spawn，不得由主代理自行扮演**，避免自我背书）。三视角分工：

| 质疑者 | 视角 | 攻击点 |
|--------|------|--------|
| **证据质疑者** | 引用的证据真在原文里吗 | 核对证据指针指向的计划章节/代码行/日志行是否真实存在、是否断章取义、是否误读时间戳/上下文 |
| **替代解释者** | 有没有别的原因能解释 | 必须提至少1个竞争假设：该结论描述的"问题/根因"是否其实是正常行为/另一模块引起/误读？能否被推翻？ |
| **充分性质疑者** | 现有证据够不够 | 结论是否被证据充分支撑？还缺什么证据才能定论？ |

### spawn 规格要求（强制，防子代理失败）

每个质疑者 spawn 时**必须**满足以下规格，否则极易失败（实测痛点：默认/误用 Explore 子代理 → 只调研不裁决 → 调研阶段耗尽额度被截断、只返回开场白）：

1. **`subagent_type: "general-purpose"`**——**禁止用 `Explore`**（Explore 是只读搜索代理，擅长定位代码、不擅长裁决判断，长指令下倾向先调研后输出、被截断在调研阶段）
2. **用 `schema` 参数强制 StructuredOutput**——子代理必须调用结构化输出工具返回如下对象，不得输出长篇自然语言调研报告：
   ```json
   {
     "verdict": "confirmed | refuted | needs_more_evidence",
     "reasoning": "裁决依据（≤200字，必须含证据回指）",
     "evidence_ref": "回指到具体 file:line / 计划章节 / 日志行",
     "competing_hypothesis": "仅替代解释者必填：≥1 个竞争假设"
   }
   ```
3. **指令开头强制约束**：「直接输出裁决，读文件范围仅限喂你的路径，不要长篇调研。先给 verdict 再给依据。」

## 输入契约（喂给每个质疑者的内容，三份相同）

- 分析对象所在文件路径（计划/代码/日志）——让质疑者自己读，**不喂主代理的推理过程**
- 待验证的清单（issue 或根因假设），含其证据指针
- 指令：「独立读文件判断每条结论，返回裁决 + 具体反驳/支持依据 + 行回指。默认怀疑，证据不足时判 `needs_more_evidence` 而非 `confirmed`。」

> **独立性硬约束**：质疑者只能拿到"文件路径 + 结论清单 + 证据指针"，严禁把主代理推理链/分析过程原样塞给质疑者当既定事实。质疑者必须自己读文件独立判断，否则对抗形同虚设。

## 零待对抗快速通道

若分析对象为 0 条需对抗项（全无问题，或全部已实证无需对抗），跳过本步骤，直接置 `adversarial_verification = null`，进入产出写入。不得为空清单 spawn 质疑者浪费资源。

## 子代理失败处理（重试→降级，绝不自演）

实测痛点：质疑者子代理偶尔失败（返回空、截断只剩开场白、schema 不符合、超时）。**禁止以"子代理调不通"为由改由主代理自演裁决**——主代理自演 = 自己提的结论自己质疑自己 = 既当运动员又当裁判，确认偏误会让它倾向确认自己的结论，对抗形同虚设（违反 anti_laziness.md 第9/14条"质疑者不得主代理自演"）。

**正确失败处理链路**（按顺序）：

1. **检测失败**：子代理返回空 / 无 schema 结构 / 无 verdict 字段 / 只剩开场白无裁决 → 判定本次 spawn 失败
2. **重试 1 次**：重新 spawn 同一质疑者，**换措辞**（强化"先给 verdict 再给依据、读文件范围仅限喂你的路径"），**换 subagent_type 兜底**（general-purpose 仍失败可换 `claude`）
3. **重试仍失败 → 诚实降级**：该条结论的对抗验证置为**未完成**，按裁决优先级的"兜底"档处理——降级为 `needs_more_evidence`，标 `[未验证-子代理对抗失败]`，计入 `pending_verification`，**绝不伪造 confirmed**
4. **留痕**：在 `adversarial_verification` 字段记录"子代理对抗失败，已重试1次仍无果，诚实降级为待验证"，附失败质疑者角色 + 失败现象

> **为什么宁可降级也不自演**：诚实承认"这条没验成"比"自己点头说过了"可信。`needs_more_evidence` 让步骤3定稿时知道这条需重点复核；伪造 `confirmed` 则把未经独立验证的结论混入定稿，埋下隐患。对抗失败是"少验了一条"，自演是"骗自己验过了"——后者危害更大。

## 主代理实证 ≠ 自演（澄清边界）

主代理用 Read/Grep 查到的**代码/日志铁证**（如"某函数实际签名是 X"、"某行日志确实存在"）属于**事实核查**，不是对抗验证，**不算自演**。这类实证结论可走"实证快速通道"直接 `confirmed` 无需 spawn 质疑者（adversarial.md 各步骤分析对象段已声明）。

**真正必须独立 spawn 对抗的**是**判断性结论**——"这个设计合理吗"、"这个根因对吗"、"这个风险够不够"——这类没有铁证、依赖独立视角的判断，主代理自演无效，必须独立 spawn。失败时按上节"子代理失败处理"降级，不得自演。

## 裁决优先级（按顺序判定，命中即定，未命中进下一档；三档都不命中走兜底）

1. **`refuted`**：≥2 个质疑者判 `refuted`，**或**替代解释者给出被证据支撑的竞争解释且证据质疑者确认原证据失效 → 撤销，记入被推翻清单，写明推翻原因
2. **`needs_more_evidence`（一票降级）**：充分性质疑者判证据不足 → 降级进 `pending_verification`，标 `[未验证-证据不足]`，不计入 confirmed 清单（证据充分性是 confirmed 的前提，享有否决权）
3. **`confirmed`**：≥2 个质疑者判 `confirmed`（且未被前两档命中）→ 保留，计入最终清单
4. **兜底 `needs_more_evidence`**：以上三档均未命中（如三态分裂 `1 confirmed + 1 refuted + 1 needs_more`、或 `2 needs_more + 1 其他` 等）→ 降级进 `pending_verification`，如实记录"对抗未收敛"

> **为什么充分性质疑者一票降级**：`confirmed` 的前提是"证据充分支撑结论"。若充分性质疑者认定证据不足，即便另外两人判 confirmed，结论也站不住——此时确认等于伪造共识。故证据充分性享有否决权。
>
> **为什么需要兜底**：三人三态裁决可能出现无任何状态≥2票的分裂组合。兜底确保这类"对抗未收敛"的结论不会因规则漏洞被误判为 confirmed，也不会被丢弃——一律降级待验证。

## 循环

分析师出结论（issue/根因假设 H + 证据指针 E + 置信度）→ 质疑者攻击 → 有实质反驳就修正/补证据/降级 → 重复直到收敛。

- **收敛判据**：所有质疑者一致认为成立 → 接受；达上限（默认4轮，可调）仍未收敛 → 输出"最强存活假设"，如实写明残余不确定性，**绝不伪造共识**；证据耗尽 → 同上诚实降级。

## 诚实底线

若 3 个质疑者对某条结论裁决严重分裂、无法形成多数意见，不得"和稀泥"默认 `confirmed`——必须降级为 `needs_more_evidence` 并在产出中如实记录"对抗未收敛"。**宁可降级，也不伪造共识。**

若证据不足以支持任何高置信结论，报告写"现有证据不足以定论，已排除 X/Y/Z，建议补充 W"。**宁可说不知道，也不编一个结论。**

## 证据回指纪律

每条 confirmed 结论必须能回指到**具体证据片段**（计划章节/代码 file:line/日志 节点+时间+原文）。做不到回指的结论不写进「核心结论」，只进「待验证假设」。

## 各步骤的分析对象

- **步骤2 review**：分析对象 = 步骤2.5 逐维审查产出的 issue（步骤2.4 实证验证失败的 issue 例外，已有铁证直接 confirmed 无需对抗）
- **入口 log**：分析对象 = 阶段2 现场还原后分析师提的根因假设 H
